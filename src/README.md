# src

`src/` 目录存放 qq-bridge 的运行时代码。

当前主入口：

- `index.js`：桥接服务启动入口

---

## 当前职责

这一层主要负责把“消息收发”与“行为决策”串起来，属于整个项目的运行核心。

### 1. NapCat / OneBot 接入

- 建立事件 WebSocket 连接
- 建立 API WebSocket 连接
- 接收 QQ 消息事件
- 调用 OneBot API 发回消息
- 维护连接状态、重连、错误日志

### 2. 基础消息处理

- 标准化 OneBot 事件
- 提取消息文本
- 区分私聊 / 群聊
- 判断是否被 @
- 识别用户、群号、消息 ID 等元信息

### 3. 白名单与过滤

- 私聊白名单
- 群聊白名单
- 忽略自身消息
- 群聊触发前置过滤

### 4. 决策层（Decision Layer）

负责判断：

- 要不要回复
- 回复模式是什么
  - 私聊回复
  - 群聊 @ 回复
  - 群聊名字点名回复
  - 群聊插话
- 为什么回复
- 当前 human state 是什么
- 应该延迟多久再回复

### 5. 回复层（Reply Layer）

负责：

- 调用 LLM 生成回复
- 保留 stub fallback
- 对文本做口语化清理
- 让输出更接近即时聊天短句
- 必要时再做兜底切分

### 6. 可观测性

- 输出终端日志
- 写入 `logs/bridge.log`
- 提供 `/health`
- 提供 `/debug/state`
- 提供 `/debug/config`

---

## 当前建议的内部结构

虽然现在代码可以先集中在 `index.js` 中快速迭代，但从职责上，可以理解成以下模块：

- **transport**
  - NapCat / OneBot 连接与收发
- **policy**
  - 白名单、触发规则、群聊策略
- **decision**
  - 该不该说、何时说、用什么模式说
- **reply**
  - 用 stub / LLM / 后续 OpenClaw 生成文本
- **formatter**
  - 短句化、分句、兜底切分
- **state**
  - 群聊历史、私聊历史、最近决策、最近发送记录、连接状态

当前不强制拆文件，但代码设计上应尽量朝这个方向组织，避免把所有逻辑混成一团。

## 备注

这层代码最重要的不是“能回复”，而是“回复行为可控、可解释、可调试”。

一个真正可长期运行的聊天桥接器，必须优先保证：

- 为什么回 / 不回是清楚的
- 什么时候回是可调的
- 群里存在感是可控的
- 内容生成失败时可以降级
- 日志能帮助快速定位问题

所以，`src/` 目录代表的不是单纯的接口转发，而是整个系统的行为核心。
