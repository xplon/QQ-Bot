# 新增阶段性架构构想随笔 (ARCH_NOTES)

*本文档替代原先已过时的 `NapCat WebUI Debug WS 发现`，由于主要通道建设均已落地，留作我们下阶段系统演进的草稿灵感本。*

## 1. 记忆与话题的交织挑战 (Memory & Threading Challenge)
根据我们的最新愿景（处理群聊中多话题并行、长期记忆）：
群聊环境是高并发、高噪音的。不同的人可能正在同时聊不同的话题：
- 用户 A 与 B 聊：“今晚吃什么？”
- 用户 C 与 D 聊：“代码这块有个 bug”

如果我们要机器人能在恰当时间插入话题（比如回答代码问题，而忽略今晚吃什么），我们需要在内存中实时提取子话题 (Sub-threads)：
- **思路**：可以尝试在每次新消息到达时，计算与活跃前置对话序列的“文本连贯度”或依靠 `reply_id` 构建有向无环图(DAG)。
- 当被触发时，仅提取同一路聚类节点或相同连通分支交给 LLM，这就实现了精准打击，**“厘清对话线”**。

## 2. “认知用户” (User Profiling)
当一个人在群里说“帮我查一下昨天我关心的那个汇率”，我们需要怎么做？
- 当前 `state.json` 只能存近期的字符串列表。
- 新架构的设想是：
  每天系统在凌晨会唤醒“总结员代理 (Summarizer Agent)”。它会遍历当天的历史日志，对每个用户的言行进行抽壳，把 `喜好`、`语言风格的描述`、`重大事件标签` 写成类似于 `profiles/123456789.json`。这个也就是**持久化的群聊和私聊画像**。

## 3. 情绪检测前置阀 (Emotion Probing Valve)
**为什么要检测情绪？** 因为如果群聊氛围不对，乱发表情包或冷笑话是不恰当的。
- 我们可以在目前的 `normalizeEvent` 和 `checkAllow` 间加入一个“环境分析探针”。如果最近的 15 条消息包含大量负面词汇或者感叹号密集，我们将群聊的一个变量 `group_vibe_score` 降为 -1，此时我们可以使 `allowInterrupt=false` 或者让提示词系统限定仅以“共情口吻”回应。

## 4. 图像识别的基建 (Vision Capability)
现有的 `cleanText(text)` 会把 `[CQ:image...]` 这类内容生硬抹去，要接入表情包，我们下一步得改变 `normalizeEvent`。可以提取图片链接并通过本地部署 VQA 或轻量级 CLIP 模型完成转译，最后喂给主控 LLM 的 Context 里应该是：
`"张三发送了一张图片：(内容描述：一只猫猫头在挠屏幕，上面有配字'可恶')"。`
这样机器人不仅听到了话语，还“看懂”了图。这是未来进阶的一大里程碑。
