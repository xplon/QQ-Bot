# 下一步行动指南 (Next Actions)

项目目前已具备较为成熟的核心通道、拦截器、回复生成逻辑及本地的持久化支持。
针对“长记忆”、“用户画像”、“情绪与话题网络”以及“多模态”等这几个宏大新目标，建议我们将工作拆分为以下几组独立且可落地的任务队列逐一击破：

---

### Action 1. 铺垫长期记忆与画像体系
**描述**：目前只是把 `state.chatHistory` 序列化为磁盘上的大型 JSON，随着记录增多必将臃肿，急需迁移升级。

*   **Task 1.1**: 将数据存储从单一大文件 `state.json` 替换为更有针对性的实体拆分（例如：拆分为 `/storage/history`、`/storage/profiles`，或引入轻量级 SQLite/NeDB）。
*   **Task 1.2**: 建立 `UserProfile` 和 `GroupState` 的对象层结构定义。
*   **Task 1.3**: 研发一个独立的定时脚本或异步任务（“后台闲时思考进程”），读取最近 100 条聊天摘要，通过 LLM 总结出特定使用者的“画像标签”，并更新覆盖到新文件下。

### Action 2. 加入消息“线索(Thread)”追踪器
**描述**：需要解决目前的“大杂烩” Context，引入对话线分流。

*   **Task 2.1**: 增强 `pushChatMessage()` 里的上下文关联。现在我们只存了时间戳、文本、用户。增加 `threadId` 和 `replyToId` 属性关系保存。
*   **Task 2.2**: 试构一套简单的聚类逻辑，或者在向大模型请求回复前，通过 LLM 判断 "给定这段近期群消息，它们属于几个讨论话题？"
*   **Task 2.3**: 重构 `buildContext()`，当针对某条群组消息（如被 @ 或点名）生成回复时，优先提取与之所属同 Topic 的群成员上下文，过滤噪音弹幕。

### Action 3. 情绪探针预处理
**描述**：给群聊策略加装“看脸色”功能。

*   **Task 3.1**: 维护一张以群号和用户 ID 关联的情感值状态表（从 -1.0 到 1.0 的值域）。
*   **Task 3.2**: 对用户的每一批发言通过廉价快速的小模型或分类器前置推断情绪极性，将打分实时更新在状态表上。
*   **Task 3.3**: 修改 `computeDelayMs` 和 `checkAllow`，若群落情感值呈“紧张争吵（Angry）”状态，触发强行静默阻断或延长缓冲保护；若是“轻松娱乐”，则提升幽默插话阈值。

### Action 4. 拦截多模态富元素
**描述**：从纯文本桥接变为图文混合桥接。

*   **Task 4.1**: 修改 `normalizeEvent()` 方法，专门保留并结构化解析包含图片占位符/URL 的内容，而不仅仅是拼接文字。
*   **Task 4.2**: 集成 Vision LLM 接口能力以响应图片信息，或对接快速的本地 OCR 提取文字。
*   **Task 4.3**: （附加探索）支持本地调用工具（类似当前的 `mcp.js`），自动回发随机图库中的表情包链接。

---
建议先行开展 **Action 1** 建立底层记忆与画像数据的地基，然后再着手 **Action 2** 处理复杂的多线程群聊消息重构。
